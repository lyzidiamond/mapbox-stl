First, what is Mapbox Studio?
Mapbox Studio is a tool for styling custom basemaps.
It comes pre-loaded with a vector tile dataset called mapbox-streets-v5, generated from a combo of OpenStreetMap and Natural Earth data!
Using a styling language called CartoCSS, you can apply style rules to the vector data to create map tiles.
Mapbox Studio is also a tool for creating your own vector tile datasets from custom sources.
Let's talk about vector tiles.



Before there were vector tiles, there were raster tiles.
(If you've seen Anatomy of a Web Map then you already know all about this, but bear with me.)
Raster tiles are 256px x 256px PNG files that are stored on the server and served to the browser on request.
Only the tiles in the current map view are requested and loaded. This makes maps super hella fast.
How are raster tiles made?
1. Use vector data (shapefile, GeoJSON, etc)
2. Style the data with a styling language called CartoCSS
3. Export data and stylesheet to a renderer (Mapnik) that creates tiled images.
The images are created at each zoom level and stored in a specific file structure as PNGs.
That makes sense, right?
Vector data + style = image with style applied to data.
(This is how TileMill worked.)
Then, using a web mapping library like Mapbox.js, you create a map container that knows when to load which tiles.
When your map needs the image tile for a particular area at a particular zoom, your map container (via the browser) requests it from the server.
Hooray! A map!
Mapbox Studio and vector tiles are only slightly different.
What's different?
1. Use vector data
2. Convert the vector data from its original format to vector tile format (mbtiles or pbf) via upload from Mapbox Studio to Mapbox.com.
3. Load the vector tile data (instead of the raw vector features) into Mapbox Studio with its dataset ID and style with CartoCSS
4. Export style to Mapbox.com.
5. When a style is requested in the browser, Mapnik takes the style, grabs the source it references, and creates image tiles ON THE FLY to serve to the browser.
Why is this beneficial?
A) Vector tiles are stored with all its attributes and can be styled with multiple styles.
B) This is a way more compact system for data storage (not having to store retina and non-retina, etc).
C) You can give Mapnik options on request (e.g. retina, multiple styles) and the tiles are created and served quickly.
D) Hosting vector data independently of the style means it can be rendered in many different ways...
... which is where the new Mapbox Studio comes in.
Some of you may have seen or heard of this crazy Mapbox GL thing.
GL is a different way of rendering data on your screen.
With both older raster tile creation and the newer vector tile creaton process, your map is requesting images -- slower.
With GL, instead of passing the images to the browser...
... you will actually pass the vector data itself to the browser...
... where the WebGL processor will render the graphics right in your browser -- clearly fastest.
This is HUGE and will change the way we deal with maps forever.
How? Some major things:
1. Fractional zoom and fading between zoom levels.
When creating vector tiles, you get to decide which datasets are stored at which whole-number zoom level.
But the data can be rendered at fractional zoom levels, because it's vector data and vectors always have good resolution.
This means awesome, smooth transitions.
2. The overlay/basemap paradigm is over.
Right now, you typically have a map tile basemap raster layer, and a vector layer on top that you can interact with via JavaScript.
Rasters always on bottom, vectors always on top.
With GL, it's all vectors, which means you can put any data in any order.
For example, a choropleth map with labels on top!
3. Super fast maps on mobile.
In addition to the WebGL processing, we also are creating native GL SDKs for mobile.
This means that, instead of requesting rasters over a phone connection (slow), users are requesting vectors (fast!) that are rendered with OpenGL right on the device.
Super fast. Super handy.
But what does this have to do with Mapbox Studio?
Let's talk about CartoCSS for a minute. What is CartoCSS, really?
It is a styling language that serves as a pre-processor for Mapnik styles.
Mapnik can read CartoCSS and translate it into style rules when it's generating raster tiles.
Mapbox Studio styling, at its core, is a visual styling interface for creating style rules for Mapnik.
When we start using GL, we are no longer using Mapnik.
So, how do styles work in GL?
Glad you asked.
[image of gl style spec json]
A pretty complicated JSON file.
CartoCSS is not a pre-processor for GL styles.
So we need something else to serve as a visual interface for styling, something that can be translated into GL styles.
That is what the new version of Mapbox Studio is setting out to do.
With the new Mapbox Studio, you will be able to create your own style JSON via a visual interface, with buttons and sliders.
This will then be translated into GL style JSON, which will be served with the vector tiles for rendering on the client-side (or native if mobile).
This is super awesome.
But it's also going to be a shift in the way we do things.
What are some of the main challenges here?
1. Moving from a raster tile code paradigm to a GL code paradigm for map interactions.
With GL vector tile rendering, you will be using Mapbox GL JS on the front-end instead of Mapbox.js.
Some things that are easy in one are hard in the other, and vice versa.
There will be lots of examples!
2. Moving from CartoCSS styling to GL styling.
There is no practical way to convert CartoCSS style sheets to GL styles.
CartoCSS has whole-number zoom-specific styling...
... while GL has fractional zoom and transitions.
That's just one complication.
3. Not every browser/system can support GL.
Fortunately, you can easily get raster tiles from your GL styles should you need them...
... but you'll have to build interactions in Mapbox.js AND Mapbox GL JS to keep things like tooltips.
It's a little bit of a different way of thinking about things, and it's confusing.
